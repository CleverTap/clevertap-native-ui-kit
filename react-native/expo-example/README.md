# `@clevertap/native-display-sdk` Expo example

This is an Expo-managed React Native example app that exercises every renderer
path in `@clevertap/native-display-sdk`. It is a sibling of the bare-RN example
at `../example/` and reuses the bare example's screens 1:1 via a Metro alias -
so changes to a screen show up in both apps.

## Why two example apps?

| | Bare example (`../example/`) | Expo example (this folder) |
|---|---|---|
| Toolchain | `react-native` CLI + Pods + Gradle | Expo CLI + `expo prebuild` |
| iOS / Android folders | Checked in | Generated by `expo prebuild` |
| CleverTap setup | Hand-edited Podfile + AndroidManifest | `@clevertap/clevertap-expo-plugin` writes them at prebuild |
| Screens | Source of truth at `./screens/` | Pulled in from `../example/screens/` via the `@bare-example/*` Metro alias |

The bare example is for clients on a vanilla RN setup. This Expo example shows
the same SDK working in the Expo workflow alongside `clevertap-expo-plugin`.

## Architecture

```
react-native/
├── src/                       ← The SDK (loaded via metro resolver)
├── example/                   ← Bare RN example
│   └── screens/               ← Single source of truth for screens
└── expo-example/              ← THIS APP
    ├── App.tsx                ← Imports screens from @bare-example/screens/*
    ├── metro.config.js        ← Aliases @bare-example/* and @clevertap/native-display-sdk
    ├── babel.config.js        ← babel-preset-expo + reanimated/plugin (last)
    ├── app.json               ← Expo config with @clevertap/clevertap-expo-plugin
    └── stubs/                 ← Null shim for missing optional peer deps
```

## Setup

### 1. Bootstrap the local plugin (one-time)

Because we install `@clevertap/clevertap-expo-plugin` from a local `file:`
path, npm runs the plugin's `prepare` script (`expo-module prepare`) inside
the plugin folder. That script needs `expo-module-scripts` from the plugin's
own `devDependencies`, AND it generates the plugin's `build/` output that
`app.plugin.js` requires at runtime. So before the example can install or
run, the plugin needs to be built once:

```bash
cd /Users/akash.malhotra/clevertap-expo-plugin
npm install
npm run build
```

You only need to redo this if you change the plugin's source.

### 2. Install JavaScript deps

```bash
cd /Users/akash.malhotra/clevertap-native-ui-kit/react-native/expo-example
npm install
```

Note: this app links local copies of three packages:

- `@clevertap/native-display-sdk` → `file:..` (the parent SDK folder)
- `clevertap-react-native` → `file:/Users/akash.malhotra/clevertap-react-native`
- `@clevertap/clevertap-expo-plugin` → `file:/Users/akash.malhotra/clevertap-expo-plugin`

If you're not the original author or those paths don't match your machine,
update them in `package.json` before installing.

### 3. Fill in CleverTap credentials

Open `app.json` and replace the `YOUR_ACCOUNT_ID` / `YOUR_ACCOUNT_TOKEN`
placeholders inside the `@clevertap/clevertap-expo-plugin` config block.
Adjust `accountRegion` if needed.

### 4. Prebuild + run

```bash
npx expo prebuild --clean
npx expo run:ios        # or
npx expo run:android
```

`expo prebuild` regenerates the `ios/` and `android/` folders from `app.json`,
running every step of `@clevertap/clevertap-expo-plugin` along the way - which
edits the Podfile, Info.plist, AndroidManifest, Gradle, and lifecycle wiring.

> [!IMPORTANT]
> **Expo Go is not supported.** `clevertap-react-native` is a bare-only native
> module, so the app must be built via `expo prebuild` + `expo run:*`, or via
> EAS Build. Trying to launch via Expo Go will fail at the
> `CleverTap.autoIntegrate()` call site.

## How the live-share works

`App.tsx` imports screens like this:

```tsx
import { CleverTapIntegrationScreen } from '@bare-example/screens/CleverTapIntegrationScreen';
```

Metro's `resolveRequest` (see `metro.config.js`) maps `@bare-example/*` to
`../example/*`. So the screen file actually loaded is
`react-native/example/screens/CleverTapIntegrationScreen.tsx` - the exact same
file the bare example loads. Edit it once, both apps update on next reload.

The screens themselves use only:

- `react`, `react-native`
- `clevertap-react-native`
- `@clevertap/native-display-sdk`
- relative imports inside `../example/` (e.g. `../testConfigs/configRegistry`)

Metro's `watchFolders` includes the bare example folder so file edits trigger
hot-reload here too.

## Optional peer deps

Every native UI peer dep declared by the SDK is installed in `package.json`,
so this example exercises every renderer code path. One deliberate omission:

- **`react-native-fast-image` is NOT installed** because its v8.6.3 peer range
  pins to `react@^17 || ^18`, which collides with React 19 that Expo SDK 55
  uses. The Expo branch of `optionalDeps.ts` picks `expo-image` first anyway,
  so dropping `react-native-fast-image` here doesn't affect any visual.

If you want to test the graceful-degradation paths in
`src/optional/optionalDeps.ts`, `npm uninstall` whichever package you want to
drop - Metro's resolver will substitute the null stub at
`stubs/optional-stub.js` and the SDK will fall back to its degraded rendering
at runtime.

## npm peer-deps note

`.npmrc` sets `legacy-peer-deps=true`. This is the same flag Expo's official
templates use - older RN-community packages often pin React/RN peer ranges
that haven't been bumped for newer SDK versions, and the flag stops `npm
install` from rejecting them. Code-level compatibility is mostly fine; it's
just the npm metadata that's stale.

## Verification checklist

After `expo run:ios|android` succeeds, walk through:

- [ ] The app boots into the same bottom-tab layout (Events / Slots / Browser
      / Bridge / Banners / Fonts) as the bare example.
- [ ] Browser tab's chip strip starts with `prod-001`...`prod-008` then
      `test-178`, `test-177`, `image-fit-test`, `test-001`, ...
- [ ] Test-014 chips render at uniform heights.
- [ ] Test-030 grid items don't overflow the right edge.
- [ ] Events tab fires a named event and shows the success log entry
      (requires real CleverTap credentials in `app.json`).

If all five pass, the SDK is verified working in Expo with no source changes.
