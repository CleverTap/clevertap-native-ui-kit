# HTML

The **HTML** element embeds arbitrary HTML inside a `WebView` (Android) or `WKWebView` (iOS). Use it for rich content the JSON schema can't express — formatted articles, embedded charts, third-party widgets, or rendering already-authored HTML email-template slices.

If the content is structured enough to express with our other elements, prefer those — HTML is heavier per impression and harder to style consistently.

## Features

- Inline HTML string (`bindings.html`) or remote URL (`bindings.url`)
- Optional JavaScript execution (off by default)
- Transparent background by default (so themed parents show through)
- Custom `baseUrl` for relative resource resolution

## JSON schema

```json
{
  "type": "element",
  "elementType": "html",
  "id": "promo",
  "bindings": {
    "html": "<h1>Hello {{userName}}</h1><p>Welcome.</p>",
    "url":  "https://example.com/promo.html"
  },
  "layout": {
    "width":  { "value": 100, "unit": "percent" },
    "height": { "value": 240, "unit": "dp" }
  },
  "htmlConfig": {
    "javascriptEnabled":   false,
    "scrollEnabled":       false,
    "transparentBackground": true,
    "baseUrl":             "https://example.com/"
  }
}
```

### Binding precedence

If both `bindings.html` and `bindings.url` are set, **`html` wins**. The URL is ignored. Choose one.

| Binding | Use when |
|---------|----------|
| `html` | The content is small / known up-front / generated by your backend. Faster — no extra HTTP request. |
| `url`  | The content is large / hosted externally / cacheable as a CDN asset. |

### `htmlConfig`

| Field | Default | Notes |
|-------|---------|-------|
| `javascriptEnabled` | `false` | Enable only if you fully trust the content. JS in a WebView is a meaningful security surface. |
| `scrollEnabled` | `false` | Lets the inner WebView scroll. Most layouts want this off so the parent container handles scroll. |
| `transparentBackground` | `true` | Lets the parent's `backgroundColor` show through behind the HTML. |
| `baseUrl` | none | Resolves relative `<img src>`, `<script src>`, `<link href>` paths inside `bindings.html`. |

## Platform parity

| Platform | Primitive | Source |
|----------|-----------|--------|
| Android | `android.webkit.WebView` | `ElementRenderer.kt:256–284` |
| iOS | `WKWebView` (wrapped in `UIViewRepresentable`) | `NativeDisplayRenderer.swift` (HtmlNode branch) |

WebView/WKWebView render almost-identical HTML, but font defaults differ (Android serif vs Helvetica). Specify CSS `font-family` in your HTML to keep cross-platform parity.

## Examples

### Inline HTML

```json
{
  "type": "element",
  "elementType": "html",
  "id": "promo",
  "bindings": {
    "html": "<style>body{font-family:-apple-system,Roboto,sans-serif;color:#fff;margin:0}</style><h2>50% off this weekend</h2>"
  },
  "layout": { "width": { "value": 100, "unit": "percent" }, "height": { "value": 80, "unit": "dp" } }
}
```

### Remote URL with JS enabled

```json
{
  "bindings": { "url": "https://example.com/widget.html" },
  "htmlConfig": { "javascriptEnabled": true, "scrollEnabled": true },
  "layout": { "width": { "value": 100, "unit": "percent" }, "height": { "value": 320, "unit": "dp" } }
}
```

## Common pitfalls

**`layout.height` MUST be explicit.** WebView doesn't expose intrinsic content height to its parent. If `height` is `wrap_content` or omitted, the WebView either collapses to 0 or expands infinitely depending on platform — both broken. Always pin `height` to a percent or fixed unit.

**JavaScript is off by default.** Pages relying on `<script>` execution silently render their no-script fallback. Set `javascriptEnabled: true` deliberately.

**Cross-origin resources** with `bindings.html` need an explicit `htmlConfig.baseUrl` that matches the resource origin, otherwise CORS / mixed-content blocks them.

**Large `bindings.html` strings** bloat the JSON payload that the SDK parses. If the content is over ~10 KB, prefer `bindings.url`.

**Tap-handling differs from native elements.** Taps inside the HTML go to the WebView, not to the parent's `actions.onClick`. You can't wrap an HTML element in a BOX with click actions and expect them to fire — the WebView eats the gesture.
